# Minimal process

[`/realize`](../realize/SKILL.md) dispatches here when a change belongs in the repository and ships as a PR but has no design left to settle: no term to coin, no domain logic to shape, no contract to pin. It keeps [full-process](../_full_process/SKILL.md)'s tail (commit, CI, the pull request, the convergence in `/parlay`) and drops only the [Design](../design/SKILL.md) pass and its panel. If real design turns out to be in play, the router sent you here in error; the honest move is full-process, not a light frame forced over it. Agent-facing prose composition never belongs here either: it takes prose-process, settled or not, because for prose the composition is itself the work to bracket.

## Stay grounded: the one thing the light path must not lose

Dropping the Design bracket is not dropping the Doctrine; a light path that shed its grounding would fall back to vanilla Claude Code behaviour. The bearing (`~/.vaudeville/doctrine/bearing/`) and the code discipline (`~/.vaudeville/doctrine/code/`) bind every line here exactly as on the heavy path, and the one that bites hardest on a light change is that you still build the smallest thing that serves the goal. Recall them as you work, and re-read from source wherever the change is not obvious.

## Shed a heavy conversation first: your call

If you arrive holding a long conversation and have not checkpointed recently, consider a bare `/checkpoint` before you implement, to carry the work into a fresh context. (A Checkpoint is a custom compaction built around the operator's turns, which it preserves, coupled to a bootstrap that resumes you into this same work afterward.) This is judgment, not a step: light work in a small context needs none. Skip it freely when that is your situation.

If you do take it, the checkpoint is **bare**: no continuation skill reloads this procedure on the far side, and the Digest carries only the operator's turns, not this skill body. So your Carryover must carry the rest of this procedure across the clear: implement, then `_tender`, and the Parlay stance below. A resumed Bob that finds none of that in its Carryover has lost the path.

## Implement

Make the change.

## Tender, and keep the convergence light

Invoke `_tender` via the Skill tool with no arguments: local CI, commit under the assignment id, push, the pull request, and the hand-off to `/parlay`. Invoking it is the last thing you do here.

Carry one stance into `/parlay`, and because `/parlay` checkpoints, into its Carryover, or the clear sheds it: hold an extra-skeptical line on Codex review suggestions. Codex tends to drive toward over-engineering, and on the light path the right solution is almost always light too. A suggestion that would turn a tiny, settled change into something heavy is suspect by default, but if it carries a point that genuinely holds, take it to the operator rather than heavying-up the change silently, and do not wave a real point away to keep things light.
