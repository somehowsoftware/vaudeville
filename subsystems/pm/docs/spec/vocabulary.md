# vaudeville-coordinator vocabulary

Terms internal to vaudeville-coordinator, referenced in this subrepo's code, tests, and spec.

## Authoring

The act of writing a new Assignment into the project tracker. Authoring is the genus act: the `/premise` path writes a Premise, `/direction` writes a [Direction](#direction), `/command` writes a [Command](#command), `/manual` writes a [Manual](#manual), and [Tangent](#tangent) writes a provisional Premise, all through the one create-plus-depend primitive. Each kind files itself with its own Type, admitted only on a [Route its kind allows](#route-constraint). `/premise` is the default door and carries the discrimination that sorts the four kinds, because a Premise and a Direction are stored alike but for a type label and nothing downstream catches a mislabel. When the Assignment is a Premise, its description takes the humble-snapshot shape defined in `~/.vaudeville/doctrine/practice/premise-frame.md`; that frame is the Premise *kind*'s, not the genus act's. Authoring sets the Route and optionally wires Depend edges to peer Assignments. The `/premise` skill accomplishes it in-thread: the current agent composes the Premise and files it directly, the deliberated posture, used when the work has been thought through and full composition is earned. An optional `--spawn` chains a Bob launch onto the filed id through the bobiverse facade. The `vv file` subcommand is the single-call primitive `/premise` routes through, defaulting to Type Premise and taking the kind on `--type`; [Tangent](#tangent)'s deterministic-capture path reuses it for the thin-signal posture.

## Awaiting Sign-off

A Command or Manual a close has freed but the operator has not yet signed off: open, every Assignment it depends on resolved, yet held outside the pickup pool until the operator's go-ahead. It is the pre-sign-off mirror of [Pickable](#pickable) (the same open-and-deps gate, differing only on whether the operator has signed off), so a freed Command or Manual is in exactly one of the two: pickable once signed off, awaiting sign-off until then. The two divide on that single line, so neither a gap nor an overlap can open. `vv unblocked-sign-off` lists the awaiting-sign-off peers a resolved Assignment has freed, one id per line, on a surface separate from `vv unblocked` so that query's auto-pickup output still feeds a spawn loop untouched. The surface only reports, never spawns: a close surfaces these for the operator's sign-off *decision* (which may be to decline) rather than letting freed operator-authored work stall unseen. A signed-off Command or Manual is [Pickable](#pickable) instead: already in the auto-pickup pool, listed by `vv unblocked`, spawned there.

## Claim

The Workflow transition Submitted → Claimed, or Returned → Claimed when a previously-returned Assignment is picked back up. The act by which a Bob takes responsibility for an Assignment. vaudeville-coordinator contributes the `vv assignment-claim` primitive that writes this transition; the policy decision of when to claim (read the Assignment, decide whether to proceed, claim only on commitment) lives in the per-Route first-turn body that vaudeville-cue renders.

## Classify

Determining which Component a captured concern belongs to (its subject's Component) by a narrow model over the register's per-Component descriptions, rather than by the agent that captured the concern. Classification takes the decision off the running agent, which would otherwise anchor on the Component its thread started in; projecting the concern's text onto a fixed register is *more* deterministic than that free choice, not less. The model only proposes: an answer is admitted as a Component when it names one already in the register, so a Component the model invents can never be filed under, and a concern that matches none aborts rather than guessing. [Tangent](#tangent) is the first and only consumer; until a second earns it, classification has no `vv` verb of its own.

## Command

The operator's settled directive: authority conferred at write time, made of ordered steps and authorization gates, executed rather than pressed, the inverse of a Premise's humble proposition, which is offered to be questioned. A Command is operator-originated by definition: settled authority has a source, and the source is the operator, so an agent files one only on the operator's explicit invocation, never on its own initiative. The authoring agent's discretion is reduced to capture: it records what the operator authorized and invents nothing, asking about anything unclear rather than guessing, because a detail the operator never asked for, filed under the operator's authority, is the failure the shape exists to prevent. A detail the operator defers to implementation time is the implementing agent's to choose, not the author's. vaudeville-coordinator contributes the `vv command` primitive: it files through the same create-plus-depend path as [Authoring](#authoring)'s `vv file`, with Type fixed to Command and Route constrained to check-in (the default) or direct. `vv command` exposes no `--spawn`: a Command carries the operator's authority, so it waits outside the pickup pool until the operator signs it off, the go-ahead that admits it for spawn (see [Pickable](#pickable)). The `/command` skill composes one in-thread; rendering a filed Command's first turn to the agent that runs it belongs to vaudeville-cue, not here.

## Decomposition

The act of cutting a settled plan-route design into the child Assignments that materialize it, plus the Depend edges among them. Where [Authoring](#authoring) composes one Assignment, Decomposition composes a coherent set, and it proceeds in **rounds** rather than flattening the whole tree at once. A round cuts the children that are implications of the meaning (the names and contracts) settled so far, and lays whatever new meaning the next round needs; a child cut against meaning not yet settled forces its picker to reinvent that meaning, and siblings drift before they integrate. Because working a round changes the world the next is cut against (the Bobs on it discover what design time could not), the rounds beyond this one are not cut now. When work remains past a round, Decomposition files a **continuation**: a plan-route child depending on the round, whose picker re-runs the pass against what the round delivered. The continuation's body carries the settled goal and a pointer to the round's delivered output, never a pre-authored breakdown of children it would cut against a world that does not exist yet. A decomposition whose early work reshapes nothing after it is a single round with no continuation.

Each child takes the kind its discretion calls for, whether a [Direction](#direction) for a settled leaf (the common case, since the design settled the goal), a [Premise](#premise) where the ends stay open, or a plan Premise for a sub-area, reached through the same `vv file` primitive [Authoring](#authoring) uses and carrying the Premise bias the `/premise` gate runs; a decomposition reaches only those agent-authored kinds, never a Command or Manual. The cut (where boundaries fall, and where a round ends) is the decisive judgment and cannot be mechanized; the dependency ordering is its residue, not a separate step. A round is a transient value held in conversation, like a design, not a persisted entity; its durable residue is Assignments, Depend edges, and any continuation. It is poured out through `vv file` in dependency order so each Depend target already exists. The `/decompose` skill runs the pass. Distinct from [Authoring](#authoring) (one Assignment, not a set) and from `/groom`, which runs the same simulate-the-picker move backward over Assignments that already exist where Decomposition runs it forward over candidates that do not.

## Direction

The Assignment whose *ends* are settled on a single named assumption (the hinge) while the *means* stay open. Where a Premise leaves the goal itself open, a Direction has made the call and hands the reader the goal, the reasoning that closed it, the one hinge it rests on, and what the call reopens to if the hinge gives. The hinge is not an acceptance criterion but the single assumption whose failure reopens the call, named so the reader can attack it; that named fault line makes a Direction *more* open to cross-examination than a Premise marked merely provisional, not less. An agent reasons its own way to a Direction as it does a Premise, so it authors one on its own initiative, unlike a [Command](#command), which only the operator originates. vaudeville-coordinator files it through the same create-plus-depend path as [Authoring](#authoring)'s `vv file`, with Type fixed to Direction and Route constrained to the kind's one legal value, check-in (see [Route Constraint](#route-constraint)). Like a Premise and unlike a Command, a Direction is spawnable: the `/direction` skill chains `vv spawn` onto the filed id when the operator asks, and the Bob that wakes presses the hinge and realizes the means. The `/direction` skill composes one in-thread.

## Manual

The Assignment that delegates no discretion: the operator keeps the wheel and releases intent live, turn by turn, rather than deciding it all before the work begins. A Bob spawned on a Manual starts primed with the deferred instructions the operator captured and holds one standing posture (proactive about understanding, not about mutating) reading and raising questions but changing nothing until the operator directs it. Like a [Command](#command), a Manual is operator-originated: an agent files one only on the operator's explicit invocation and captures the deferred instructions without inventing detail. vaudeville-coordinator files it through the same create-plus-depend path as [Authoring](#authoring)'s `vv file`, with Type fixed to Manual and no Route at all: a Route classifies a conversation in advance, and a Manual expects none, since its intent arrives turn by turn (see [Route Constraint](#route-constraint)). Filing is decoupled from spawning, as with a Command: the `/manual` skill chains no spawn, and a Manual waits outside the pickup pool until the operator signs it off (see [Pickable](#pickable)), then spawns and drives it. The `/manual` skill composes one in-thread.

## Pickable

An Assignment a Bob may claim now: its Workflow is Submitted (never yet claimed) or Returned (tried and handed back), it is not yet resolved, every Assignment it depends on is resolved, and its kind has been cleared for autonomous pickup. Pickability gates the *pool* (what `vv available` and the auto-spawn loop draw from) not the explicit `vv spawn <id>`, which an authoring skill's `--spawn` reaches directly. The kind clearance is the operator's [sign-off](#command): a Premise or Direction is always cleared, since an agent authors it; a Command or Manual carries the operator's authority and is cleared only once the operator has signed it off. A freed Command or Manual not yet signed off is not pickable but [Awaiting Sign-off](#awaiting-sign-off): the pre-sign-off mirror of this rule, the two dividing on the single sign-off line so a freed Assignment of those kinds falls in exactly one. `vv available` lists the pickable Assignments in a Component; `vv unblocked` reports the peers a just-resolved Assignment has made pickable. The rule has one definition every reader shares, so they cannot disagree about what "ready to pick up" means.

Clearance is one sense of the sign-off mark, not its whole meaning. Set on a Premise or Direction, which is already cleared, sign-off gates nothing; it records that the operator has personally seen the Assignment and endorses its framing, a mark vaudeville-cue surfaces in the picker's first turn. So the one "Signed off" bit reads by kind: a pool gate for the kinds the operator authors, and an endorsement the brief carries for the kinds an agent authors. The endorsement leaves pickability untouched, which is why it changes nothing in this rule or its [Awaiting Sign-off](#awaiting-sign-off) mirror.

## Resolve

The resolving Assignment-state transition, in two dispositions: deliver (the deliverable shipped) and abandon (the Assignment should not exist anymore). Each moves the Assignment to a resolved State and records a reason comment, leaving any worktree untouched. vaudeville-coordinator contributes the `vv resolve <delivered|abandoned>` primitive over vaudeville-core's Delivered and Abandoned profiles (vaudeville-core, despite the name, is not a shared kernel but Vaudeville's outer anticorruption layer over external systems); which disposition, and whether to resolve at all, is the closeout skill's call. *Resolve* names the lifecycle's resolving close (the Assignment reaching its resolved terminus), not a programming step.

## Return

The non-resolving Assignment-state transition: a Bob attempted the Assignment, stopped partway, and hands it back rather than finishing. Returning moves the Assignment to State Active / Workflow Returned, records a return note, and unassigns it so the Assignment re-enters the pickup pool with its partial trail visible to the next picker, without tearing down any worktree. vaudeville-coordinator contributes the `vv return` primitive over vaudeville-core's Returned profile.

## Route Constraint

The (kind → permitted routes) legality rule: vaudeville-core publishes the map (`PERMITTED_ROUTES`), where a Premise takes check-in/plan/explore, a Direction only check-in, a Command check-in/direct, and a Manual no Route at all, and this context enforces it at the authoring boundary, reading the one published definition instead of re-deriving it. `vv file` and `vv assignment-create` reject an inadmissible (kind, route) before the Assignment is created; the hand-written per-Command route list this once kept is retired in favour of the published map. Enforcement is an authoring gate (the same seam the spawn side guards) so it sits at the create-adapters and Assignment creation itself stays pure policy, ungated when a caller reaches it directly. One carve-out vaudeville-core's `route_permitted` predicate gets wrong: it reports every Manual as impermissible because a Manual permits no Route, yet a correctly-routeless Manual is valid, and admitting it is the consumer's call vaudeville-core defers, so this context reads the map rather than the predicate's False.

## Tangent

A provisional Premise filed by deterministic capture rather than by [Authoring](#authoring)'s composition. A tangent is a side-concern the operator sets aside without working it out; because there is nothing yet to author, only to capture, `vv tangent` fills a fixed set of fields (a title, the operator's prompt verbatim, the possibly-relevant context, and optional Depend ids) and composes the body from them with no agent discretion. The body marks the Premise provisional and reproduces the verbatim prompt as the fixed source every other field is checked against; it has no slot for a remedy or acceptance criterion, so prescriptive content has no place to go. The Component the tangent files under is not the running agent's to choose either: absent an explicit `--project` override, it is [classified](#classify) from the concern's subject, so a tangent captured in one Component's thread files under the Component its subject belongs to rather than the originating thread's. The `/tangent` skill fills the form in-thread (the thin-signal posture, the operator's classifier for "I have not worked this out, capture it provisionally") with an optional `--spawn` that mirrors `/premise`'s. `vv tangent` files through the same create-plus-depend path as `vv file`, with Route fixed to `check-in`; it differs from the composed path only in that the body is captured deterministically and the Component classified, instead of both composed with discretion.

## Unblocked

The peers a resolved Assignment has freed for pickup. When an Assignment resolves, the peers that Depend on it may become [Pickable](#pickable). `vv unblocked` is the pure graph query that lists them, one id per line; it refuses an Assignment that is not yet resolved, since an in-flight Assignment has unblocked nothing. The query only reports; a closeout skill pipes its output into a spawn to act on it. A resolved Assignment can also free a Command or Manual that awaits the operator's sign-off, reported on a separate surface so this query's output stays the auto-pickup pool the spawn loop consumes; see [Awaiting Sign-off](#awaiting-sign-off).
