# Vaudeville is its own prototypical tenant

Vaudeville-the-project is itself a [Tenant](doctrine/vocabulary.md#tenant): it builds the next version of the framework while running on the most recent release of the framework. The show runs on itself. 

The page proves it. This essay began as a [Premise](doctrine/vocabulary.md#premise) (a proposition that the human canon ought to have one place that rips the curtain), was claimed by a [Bob](doctrine/vocabulary.md#bob) spawned into its own worktree, written against the live code it describes, and rendered into the tree you are browsing. What you are reading is not a description of the process from outside it; it is one of the process's own outputs, sitting in the published constellation beside everything else the process produced.

## The constellation

Vaudeville-the-project is not one repository. It is a federation of [Components](doctrine/vocabulary.md#component), each realized as its own git repository: the lifecycle and machinery pieces are [Contexts](doctrine/vocabulary.md#context), each a jurisdiction of meaning with its own vocabulary, its own types, its own architecture, and the doctrine that spans them all is a [Resource](doctrine/vocabulary.md#resource). The tree rendered around this page *is* that federation, gathered into one place for reading: `subsystems/` for the lifecycle contributors, `machinery/` for the parts that assemble and underpin them, and `doctrine/` for what every agent primes against. Walk it.

The three contributors under `subsystems/` are the lifecycle a piece of work travels, in the order it travels it: the same arc the [lifecycle diagram](diagrams/assignment-bob-lifecycle.svg) draws.

- **Authoring** is [`subsystems/pm`](subsystems/pm). It owns the act of writing an assignment into the tracker. That means composing the description each kind calls for (a Premise's humble snapshot, a Command's ordered steps and gates), setting the kind and the Route it admits, and wiring the dependency edges to peers. It also owns the state transitions that move an assignment through its life without tearing down a worktree. Its [spec](subsystems/pm/docs/spec/pm.md) is the place to learn what authoring actually entails; this paragraph is an introduction, not a substitute for it.
- **Prep** is [`subsystems/cue`](subsystems/cue). When a Bob is spawned, cue produces the first turn it reads: it fetches the assignment and renders the [Brief](doctrine/vocabulary.md#brief) from the framing its Route calls for, or the standing posture a routeless Manual carries instead. Which framings exist, and how each is shaped, is the kind of essential detail its [spec](subsystems/cue/docs/spec/cue.md) makes precise and this tour will not.
- **Work** is [`subsystems/bobiverse`](subsystems/bobiverse). It manages the lifecycle of a Bob from spawn to teardown: the verbs that move it (spawn, realize, parlay, closeout), the Checkpoint by which a Bob sheds an oversized conversation and resumes its own work in a fresh context, and the Prime that builds the per-repository [Foundation](doctrine/vocabulary.md#foundation) each spawn forks from. Read its [spec](subsystems/bobiverse/docs/spec.md) for the boundaries it draws against the other three.

Under `machinery/` are the two contributors a reader does not act through but the system rests on.

- [`machinery/ringmaster`](machinery/ringmaster) is the integrator: the assembly layer that turns the separate contributor repositories into one deployed scaffold. It is the subject of the next section; its [architecture](machinery/ringmaster/docs/spec/architecture.md) and [vocabulary](machinery/ringmaster/docs/spec/ul.md) specs are where the assembly is defined.
- [`machinery/core`](machinery/core) is the anti-corruption layer over the foreign systems Vaudeville runs against: the project tracker, git hosting, and the host config file. It carries the cross-context vocabulary as actual value types, and every other contributor reads and writes Assignments through core and never names the tracker product directly. It ships no skills and no command-line verbs of its own; it is the boundary the rest import. Its [spec](machinery/core/docs/spec.md) states the discipline that keeps that boundary's surface conservative.

And `doctrine/` is the [universal Doctrine](doctrine/vocabulary.md#doctrine): the one contributor that is pure framing rather than code. It is two trees for two readers: the operative tree an agent primes against, compressed hard because every token in it is paid on every turn of every Bob, and the human canon you are reading now, under no such pressure. The essays differ; the [vocabulary](doctrine/vocabulary.md) they share is the single source of truth both draw on.

## How Ringmaster assembles them

The contributors are separate repositories that never import one another's code; at runtime they reach each other only through a single deployed command-line facade, the `vv` you would type. Producing that facade (and the whole installed scaffold around it) is Ringmaster's job, and it is worth seeing at orientation depth because it is what makes a federation behave as one framework.

A deploy is a session. Ringmaster fresh-clones each contributor's `origin/main`, stages the assembled result as a release candidate the operator audits and smoke-tests, and (only if that smoke clears without a single hot-fix) builds the artifact and lets it install itself. The discipline is strict on purpose: any hot-fix at all blocks the install until the fix is productionized as a reviewed, merged pull request and re-pulled into a fresh clone, so the installed scaffold only ever reflects code that was actually reviewed. When a version is published, the same build cuts a release on the Published Home (the repository this tree lives in) for tenants to install from, and commits this readable rendering of the source beside it.

That rendering is what you have been walking. It closes the loop the second section opened: the constellation around this page is the **Exposition**, the for-reading copy Ringmaster commits when it publishes a version, and this essay is one file in it. Two things follow, and they matter enough to state plainly. First, the links above into `subsystems/`, `machinery/`, and `doctrine/` resolve here, in the published tree, because here is where the federation is gathered into one place. Read the source repositories directly and those paths point at nothing, because there the contributors are scattered across separate repositories. Second, **this tree is a reading copy, not the build input.** It is curated for legibility and is neither complete nor buildable; the authoritative record of what shipped is [`provenance.toml`](provenance.toml) at the root, which pins the exact commit of every contributor and the integrator version that assembled them. When this prose and that manifest ever disagree about what a release contains, the manifest is right and this is a story about it.

## If you are sizing this up for your own work

If you came here evaluating Vaudeville (as a person deciding whether to adopt it, or as an agent reasoning about how to structure a greenfield system or migrate an existing one), you have now seen the shape of the answer rather than a promise of one. The federation you toured is what "decompose the system into Contexts, each a Component the machinery can grip" looks like once it is real, and the assembly you watched is what holds the pieces in one coherent framework while each evolves on its own. The analogy to your own system is yours to draw.
